## What Is a Use Case?

A **Use Case** is a written description of how a user (or system) interacts with software to accomplish a specific goal. It bridges the gap between the business side — which knows *what* needs to happen — and the development team, which needs to know *exactly* how the system should behave.

Use cases are the primary way business stakeholders communicate requirements without needing to write code. A well-written use case answers three questions:

> **Who** is using the system? · **What** do they want to accomplish? · **How** does the system respond at each step — including when things go wrong?

Think of a use case as a screenplay: it describes the actors, the scene, and all the possible ways the story can unfold — including the unexpected twists.

## Key Components of a Use Case

Every use case should include the following building blocks. Some may be brief; others require detailed attention depending on complexity.

| # | Component | Description |
|---|-----------|-------------|
| 01 | **Use Case ID & Name** | A unique identifier and a short, verb-noun name that describes the action. Example: *UC-04 · Assign Technician to Service Zone* |
| 02 | **Actors** | Who (or what) participates? *Primary actor* initiates the use case; *secondary actors* support it. Can be people, roles, or other systems. |
| 03 | **Goal / Purpose** | One or two sentences stating what the primary actor is trying to achieve and why it matters to the business. |
| 04 | **Preconditions** | What must already be true *before* this use case can begin? These are facts the system can rely on. |
| 05 | **Trigger** | The specific event that starts the use case — a button click, a scheduled job, an incoming data feed, etc. |
| 06 | **Main Success Flow** | The step-by-step "happy path" — numbered actions that describe the normal, successful interaction between actor and system. |
| 07 | **Alternative & Exception Flows** | What happens when the happy path veers off? Captures edge cases, errors, and alternative routes that developers must handle. |
| 08 | **Postconditions** | What is the state of the system *after* successful completion? What has changed? |
| 09 | **Business Rules** | Constraints or policies that must be respected — e.g., "A technician cannot be assigned to more than 8 jobs per day." |
| 10 | **Non-Functional Requirements** | Performance, security, or usability expectations — e.g., "Map must refresh within 3 seconds" or "Must comply with CPNI regulations." |
| 11 | **Priority & Frequency** | How critical is this use case? How often will it be executed? This helps the dev team plan their work. |
| 12 | **Open Issues / Assumptions** | Document anything that is still unknown or assumed. This prevents silent misunderstandings between business and tech teams. |

## Tips & Cautions for Business Writers

**Write from the user's perspective.** Use active voice: "The dispatcher selects a service zone" — not "The service zone is selected." Developers need to know *who* acts first.

**One goal per use case.** Resist combining multiple goals into one document. If you find yourself writing "and then the user can also…" — that's probably a second use case.

**Name your actors precisely.** Don't say "the user." Say "the Field Dispatcher," "the Technician," or "the Billing System." Precision eliminates ambiguity.

**Think about what goes wrong.** The alternative flows are often more valuable to developers than the happy path. What if the map API is unavailable? What if a technician is already at capacity?

> **Don't specify the technology.** Your job as a business writer is to describe *what* the system must do, not *how* it does it. Avoid phrases like "store in a SQL database" or "use a React component." That's the development team's decision.

> **Avoid vague success criteria.** "The system should be fast" is not a requirement. "The route optimization result must be returned within 4 seconds" is. Make every requirement testable — if you can't write a test for it, it's probably too vague.

> **Don't skip the business rules.** Developers will make assumptions to fill in gaps you leave. If your company has policies, regulations, or constraints (union rules, FCC regulations, SLA commitments), document them explicitly. Unwritten rules become expensive bugs.
